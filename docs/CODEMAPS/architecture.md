# Architecture

## System Overview

Multi-agent research platform using Google ADK + LangSmith evals. User submits query → Orchestrator (ADK Agent) routes to 3 RemoteA2aAgent sub-agents via A2A protocol → agents invoke MCP tools → synthesized answer returned → eval pipeline measures faithfulness, relevance, citation accuracy.

```
User Query (POST /research)
    │
    ▼
Orchestrator (port 8000, FastAPI)
    │ ADK Runner + InMemorySessionService
    │ Agent "research_coordinator" (LiteLlm gpt-4o-mini)
    │ Instruction: route to web_research / rag_lookup / summariser
    │
    ├─ A2A/JSON-RPC 2.0 ──────► web_research (port 8001, ADK Agent + to_a2a)
    │                              │ CrewAI agent + MCP client
    │                              ▼ MCP HTTP → web_search tool (port 9001, Tavily)
    │
    ├─ A2A/JSON-RPC 2.0 ──────► rag_lookup (port 8002, ADK Agent + to_a2a)
    │                              │ LlamaIndex agent + MCP client
    │                              ▼ MCP HTTP → vector_db tool (port 9002, Qdrant)
    │
    └─ A2A/JSON-RPC 2.0 ──────► summariser (port 8003, ADK Agent + to_a2a)
                                   │ Anthropic SDK agent + MCP clients
                                   ├─► file_reader tool (port 9003)
                                   └─► citation_checker tool (port 9004)

    Agent Cards: /.well-known/agent-card.json (auto-generated by to_a2a)
    
    ▼
ResearchResponse {answer, sources, route, retrieved_context}
    │
    ▼
evals/ pipeline (LangSmith)
    ├─ run_evals.py: Call orchestrator with dataset Q&A pairs
    ├─ faithfulness_evaluator: Grounds answer in retrieved context
    ├─ relevance_evaluator: Addresses query + covers required topics
    └─ citation_accuracy_evaluator: Source references present & integrated
```

## Key Design Decisions (Week 4+5)

**ADK-based multi-agent**: Orchestrator and all sub-agents use Google ADK `Agent` class with LiteLlm router. Routing logic is LLM-driven via agent instruction, not hardcoded.

**to_a2a() wrapper**: Each sub-agent exposes A2A protocol via `to_a2a(agent, host, port)`:
- Auto-generates agent card at `/.well-known/agent-card.json`
- Handles JSON-RPC 2.0 message passing
- Wrapped by RemoteA2aAgent in orchestrator

**InMemorySessionService**: Orchestrator manages user/session context across requests. Sessions are cleaned up after each request to prevent memory accumulation.

**LangSmith evals** (Week 5): New `evals/` package measures:
- **faithfulness**: Claims grounded in retrieved context
- **relevance**: Answer addresses query + covers required topics
- **citation_accuracy**: Sources present and integrated naturally

**Structured synthesis format**: Coordinator instruction passes data to summariser via delimited format:
```
QUERY: <original question>
WEB_FINDINGS: <web_research output>
RAG_FINDINGS: <rag_lookup output>
```

## Request Flow (Example)

1. **User query**: `POST /research` with `{"query": "..."}`
2. **Orchestrator receives**: FastAPI extracts query, calls `_get_runner()` (lazy init)
3. **Runner.run_async()** streams events:
   - Coordinator (research_coordinator agent) processes query
   - Decides routing: web_research → rag_lookup → summariser based on instruction
   - Calls RemoteA2aAgent sub-agents via `/.well-known/agent-card.json`
   - Each sub-agent invokes MCP tools (via FastMCP HTTP client)
   - Results collected in `retrieved_context` list
4. **Session cleanup**: `await _session_service.delete_session()`
5. **Final response**: Extracts final_response event, returns `ResearchResponse` JSON
